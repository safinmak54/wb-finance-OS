How should we build this?

Three approaches to implementing full COO parity — pick the one that fits best

A
Sprint Sequence RECOMMENDED
Foundation → Report Depth → New Tabs. Each sprint pushes to QA independently.
Build in strict order: fix global infrastructure first (admin, filter bar, topbar KPIs), then deepen every report page, then build the new tabs (AP, Reconciliation) with Supabase. Three separate QA pushes — each is reviewable on its own.
Sprint 1 — Foundation
Admin role
Global filter bar
Topbar KPIs
Import modal
Sprint 2 — Reports
P&L 5 views + charts
Balance Sheet 2-col
Cash Flow GAAP
Ratios charts
Sprint 3 — New Tabs
AP/Payables tab
Reconciliation full
CFO Notes 6-notes
Sales + Product Mix
Pros
Clean incremental review · Foundation touches every page once · Lower risk per sprint
Cons
New tabs (AP, Recon) land last — CPA/COO waits longest for them
B
Schema-First FASTEST VALUE
New Supabase tables first, then all features together in one big build.
Create ap_items and reconciliation_matches tables first, seed them with WB data, then build AP and Reconciliation before touching the existing report pages. Existing pages get their upgrades in a single second pass covering everything.
Pros
Gets real AP/Recon data live early · Schema migration done before UI depends on it
Cons
Report upgrades all land at once — harder to review a single large PR
C
Page-by-Page HIGHEST RISK
Complete each page to full COO parity before moving on.
P&L to full parity → Balance Sheet → Cash Flow → Ratios → CFO Notes → AP → Reconciliation → Sales → Product Mix → Cash Forecast. Admin login and global filter bar come last as polish. Each page is a standalone commit.
Pros
Each page is clearly "done" before moving on · Easy to stop mid-way and ship
Cons
Admin login deferred · Global filter bar built 9 pages in · Inconsistent state during review
